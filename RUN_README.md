## 2. How Training Works — Complete Overview

### 2.1 Model

**`unsloth/Qwen2.5-7B-Instruct`** loaded with:
- **4-bit QLoRA** via Unsloth (`load_in_4bit=True`) → ~5 GB VRAM base
- **LoRA adapters** on all attention + MLP projection layers (`r=8`, `lora_alpha=16`)
- **Gradient checkpointing** with Unsloth's patched implementation (2× speed)
- **bfloat16** precision (native on A4500 Ampere architecture)

Estimated VRAM breakdown for A4500 20 GB:
```
Base model (4-bit):      ~5.0 GB
LoRA adapters (r=8):     ~0.3 GB   ← halved from r=16
Activations + KV cache:  ~4.5 GB   ← smaller due to max_seq_len=1536
Group buffer (8 seqs):   ~4.0 GB   ← smaller due to max_new_tokens=128
Optimizer states:        ~2.0 GB
Overhead:                ~1.5 GB
                        ─────────
Total:                  ~17.3 GB  ✓ fits with ~2.7 GB headroom
```

### 2.2 Algorithm: GRPO (Group Relative Policy Optimisation)

GRPO works by:
1. For each training prompt, generating a **group** of `G=8` completions
2. Computing a **reward** for each completion
3. Using the **relative advantage** within the group as the RL signal
4. Updating the policy to increase probability of high-reward completions

This is more sample-efficient than PPO for this task because it avoids a separate value-network.

### 2.3 Prompt Format

Every task is converted to a **single-turn ChatML prompt**:

```
<|im_start|>system
You are a misinformation forensics expert...
<|im_end|>
<|im_start|>user
CLAIM TO INVESTIGATE: "..."

CORPUS (N documents, sorted chronologically):
[DOC-XXXX] Official (Tier 1) | Source: ... | Date: ...
Title: ...
Content...
────────────────────────────────────────────────────────────
...
<|im_end|>
<|im_start|>assistant
```

The model must respond with **only valid JSON**:
```json
{
  "verdict": "true|false|misleading|unverifiable",
  "mutation_type": "distortion|fabrication|omission|context_shift|none",
  "mutation_doc_id": "DOC-XXXX or null",
  "provenance_chain": ["DOC-XXXX", ...],
  "confidence": 0.0
}
```

### 2.4 Reward Function (8 components — aligned with UnifiedGrader)

| Component | Weight | What it measures |
|-----------|--------|-----------------|
| Format validity | +0.05 | Valid JSON with all 4 required fields |
| Verdict accuracy | +0.30 | Correct true/false/misleading/unverifiable |
| Mutation type | +0.18 | Correct distortion/fabrication/omission/context_shift/none |
| Mutation point | +0.18 | Correct first-mutation document (half credit for ±1 adjacent) |
| Provenance F1 | +0.18 | F1 score of predicted vs ground-truth provenance chain |
| Source reliability | +0.11 | Weighted tier quality of cited documents |
| Hallucination penalty | −0.12 | Fabricated DOC IDs not in the corpus (0.25 per doc, capped at 1.0) |
| Brier penalty | −0.08 | Overconfident wrong answers penalised extra (1.5× if conf > 0.7) |

**Maximum total:** 1.0 | **Minimum:** −0.20

### 2.5 Three-Phase Curriculum

```
Steps   0 – 148   (0–37%)   Phase 1 — Easy only
Steps 148 – 300   (37–75%)  Phase 2 — Easy + Medium mix
Steps 300 – 400   (75–100%) Phase 3 — All difficulties (Easy+Medium+Hard)
```

The trainer rebuilds the dataset at each phase boundary by calling `trainer.train()` again with `max_steps` set to the phase boundary.

### 2.6 Data Flow

```
data/tasks/generated/*.json   (3000 tasks on disk)
           │
           ▼
load_static_tasks("easy"|"medium"|"hard")
           │
           ▼  (n_repeats=8 — each task repeated 8 times for GRPO groups)
build_training_dataset()  →  HuggingFace Dataset
           │
           ▼
GRPOTrainer.train()
  └─ generates 8 completions per prompt (group_size=8)
  └─ calls reward_fn_wrapper() on all 8 completions
  └─ computes GRPO advantage and updates LoRA weights
  └─ logs to training_logs/reward_log.json every 25 steps
```

### Step 0 (REQUIRED): SFT Warm-Up

**Why?** Without SFT, GRPO wastes 50–80% of Phase 1 because the model invents its own JSON fields (`distorted_value`, `mutant_doc_id`, etc.) instead of using the required schema (`verdict`, `mutation_type`, `mutation_doc_id`, `confidence`). A single system prompt is not enough — the model's pre-training instinct overrides it. SFT burns the exact field names into the model's weights by showing it hundreds of concrete examples.

```bash
python training/sft_warmup.py \
  --model unsloth/Qwen2.5-7B-Instruct \
  --n-examples 500 \
  --output ./chronoveritas-sft \
  --epochs 3 \
  --seed 42
```

> **500 examples × 3 epochs = 1,500 format demonstrations.** This ensures near-zero `missing_fields` errors when GRPO begins.

This saves LoRA adapters to `./chronoveritas-sft/`. Then pass that as the starting model to GRPO:

```bash
# After SFT warm-up, use the SFT checkpoint as the GRPO starting point:
--model ./chronoveritas-sft
```

> ⚠️ **Do NOT skip this step.** Without SFT, the first 100+ GRPO steps produce only `-0.10` / `-0.15` rewards due to format errors, wasting GPU time and money.

## 6. Key Arguments & A4500-Optimised Defaults

| Argument | Default | A4500 Notes |
|----------|---------|-------------|
| `--model` | `unsloth/Qwen2.5-7B-Instruct` | 7B in 4-bit = ~5 GB, safe headroom |
| `--group-size` | `8` | 8 completions/prompt. **Do not increase** — 16 will OOM |
| `--steps` | `400` | ~1.5–2 h on A4500. Use `--steps 50` for a quick test |
| `--lr` | `5e-5` | Conservative for stability. Can try `1e-4` if reward plateaus |
| `--seed` | `42` | Reproducibility |
| `--difficulty` | `curriculum` | Uses all 3 phases |
| `--output-dir` | `./chronoveritas-fact-checker` | LoRA adapters saved here |
| `--log-dir` | `./training_logs` | `reward_log.json` + `reward_log.csv` |

**Fixed in `GRPOConfig` (inside `train_grpo.py`) — optimised for A4500:**

| Parameter | Value | Reason |
|-----------|-------|--------|
| `per_device_train_batch_size` | `1` | VRAM constrained |
| `gradient_accumulation_steps` | `2` | Effective batch=2; faster than 4 with minimal quality loss |
| `max_new_tokens` | `128` | JSON responses rarely exceed 100 tokens; was 256 |
| `max_seq_length` (model) | `1536` | Fits all prompts; saves KV-cache vs 2048 |
| `bf16` | `True` | Native A4500/Ampere |
| `warmup_ratio` | `0.03` | Fewer warmup steps = more useful RL steps |
| `dataloader_num_workers` | `0` | Avoids shared-memory issues |
| `use_gradient_checkpointing` | `"unsloth"` | 2× memory savings |
| `load_in_4bit` | `True` | QLoRA — essential for fitting 7B |
| LoRA `r` | `8` | Halved from 16; faster backward, ample capacity for this task |
| LoRA `lora_alpha` | `16` | Effective scale = alpha/r = 2 |
| LoRA `target_modules` | All attention + MLP | Full parameter coverage |

## 7. Reward Function — What the Model Learns

```
reward_fn_wrapper(prompts, completions, ground_truth_json)
        │
        ▼
extract_json_safe(completion)   ← tries 3 regex patterns
        │
        ▼
compute_reward(completion, ground_truth)
  ├── [GATE] Invalid JSON          → return -0.15
  ├── [GATE] Missing required keys → return -0.10
  ├── Component 1: format            +0.05  (valid JSON with all 4 fields)
  ├── Component 2: verdict           +0.30  (exact string match)
  ├── Component 3: mutation_type     +0.18  (exact string match)
  ├── Component 4: mutation_point    +0.18  (exact doc ID; half credit if adjacent)
  ├── Component 5: provenance F1     +0.18  (F1 of predicted vs GT provenance chain)
  ├── Component 6: source_reliability +0.11  (weighted tier quality of cited docs)
  ├── Penalty 1: hallucination       -0.12  (0.25 per fabricated doc, capped at 1.0)
  └── Penalty 2: brier_penalty       -0.08  (overconfidence penalty, 1.5× if conf>0.7)
```

The model is **not given the correct answer** in the prompt. It must reason from the documents.

## 8. Curriculum Phases

```
Phase 1 (steps   0–148):
  Tasks:   Easy only (EAS-* tasks)
  Goal:    Learn to output valid JSON + get verdict right
  Dataset: build_training_dataset(tasks_easy, n_repeats=8)

Phase 2 (steps 148–300):
  Tasks:   Easy + ⅓ of Medium
  Goal:    Learn mutation_type discrimination
  Dataset: tasks_easy + medium[:len(medium)//3]

Phase 3 (steps 300–400):
  Tasks:   Easy + Medium + Hard
  Goal:    Learn provenance_chain + hard reasoning
  Dataset: tasks_easy + tasks_medium + tasks_hard
```

The curriculum manager (`CurriculumManager`) drives this automatically when `--difficulty curriculum` is passed. The trainer calls `trainer.train()` three times, updating `trainer.train_dataset` and `trainer.args.max_steps` between phases.

## 10. Troubleshooting

### CUDA Out of Memory

```bash
# Try reducing group size (fewer completions per step)
python train_grpo.py --group-size 4 ...

# Or cap max new tokens further (reduces GRPO generation buffer)
# Edit train_grpo.py: max_new_tokens=64

# Free GPU memory before starting (find which PIDs hold the GPU)
sudo fuser -v /dev/nvidia0
nvidia-smi
```

> The current defaults (r=8, max_new_tokens=128, max_seq_len=1536, grad_accum=2) are tuned to stay ~17 GB on A4500. If you still hit OOM, reduce `--group-size` to 4 first.

### `ImportError: unsloth not found`

You are in the wrong environment:
```bash
conda activate chronoveritas-train
pip show unsloth
```

If missing: `pip install unsloth==2024.9.post4`

### `No tasks available` error

```bash
ls data/tasks/generated/ | wc -l    # should be ~3000
# If 0, regenerate:
conda activate chronoveritas-train
python generate_data.py --easy 100 --medium 80 --hard 50
```

Or use `--generate-tasks` flag to let the training script generate tasks inline:
```bash
python train_grpo.py --generate-tasks --difficulty curriculum --steps 400
```

### Training reward stuck near 0.05

The model is outputting only format-valid JSON but getting everything else wrong. Solutions:
1. Run SFT warm-up first (Step 0 above) to teach the answer schema
2. Increase `--lr` to `1e-4`
3. Check that `ground_truth` in the task JSON contains all required fields

### `ModuleNotFoundError: agents.task_bank`

You must run from inside `Meta_OpenEnv/`:
```bash
cd /home/shashank/metaRLDowngraded/Meta_OpenEnv
python train_grpo.py ...
```

### Verify training is actually running (not frozen)

```bash
# Should update every ~30–60 seconds on A4500
watch -n 10 "stat -c '%y' training_logs/reward_log.csv 2>/dev/null || echo 'No log yet'"
```

### `AttributeError: module 'torch' has no attribute 'int1'` (torchao crash)

This happens when `torchao>=0.5` is installed — it requires `torch>=2.6` for `torch.int1`, but this project uses `torch==2.4.0+cu121`. The crash occurs inside peft's LoRA dispatcher even though we don't use torchao for quantization (we use bitsandbytes).

```bash
# Fix: simply remove torchao — bitsandbytes handles all 4-bit QLoRA
/home/shashank/miniconda3/envs/chronoveritas-train/bin/pip uninstall torchao -y

# Verify it's gone
python -c "import torchao" 2>&1 | grep -c "ModuleNotFoundError" && echo "✓ torchao removed"
```

> **Prevention:** torchao is sometimes pulled in as a dep by newer unsloth_zoo. After any `pip install unsloth_zoo`, run `pip uninstall torchao -y` if you're on torch 2.4.x.
